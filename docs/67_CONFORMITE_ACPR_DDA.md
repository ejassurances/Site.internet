PARTIE 1 — Philosophie de la conformité
1. Objet du document

Ce document définit les principes de conformité de la plateforme EJ Assurances.

Il constitue la référence fonctionnelle concernant :

les exigences ACPR ;
les exigences DDA ;
le RGPD ;
la traçabilité ;
le devoir de conseil ;
le recueil des besoins ;
les preuves réglementaires.

Toutes les fonctionnalités de la plateforme devront respecter les règles définies dans ce document.

2. Philosophie générale

La conformité ne constitue pas un domaine indépendant.

Elle accompagne chaque action réalisée dans la plateforme.

L'objectif n'est pas de produire davantage de documents.

L'objectif est de démontrer que le cabinet a correctement exercé son devoir de conseil.

La conformité est donc intégrée nativement au fonctionnement du CRM.

3. Une conformité invisible pour le conseiller

Le conseiller ne doit jamais avoir l'impression de remplir des obligations administratives.

Il doit simplement accompagner son client.

La plateforme se charge automatiquement :

d'enregistrer les preuves ;
de générer les documents ;
d'historiser les décisions ;
de vérifier les éléments obligatoires ;
d'alerter en cas d'anomalie.

L'utilisateur travaille.

La plateforme construit le dossier réglementaire.

4. Le Projet comme support réglementaire

Toutes les obligations réglementaires sont rattachées au Projet.

Le Projet devient le dossier réglementaire complet.

Il contient notamment :

le recueil des besoins ;
le DER ;
la Lettre de mission ;
les documents remis ;
les échanges ;
les analyses ;
le devoir de conseil ;
les signatures ;
les contrats.

Toutes les preuves sont regroupées au même endroit.

5. Les principes fondamentaux

La conformité repose sur plusieurs principes.

Traçabilité

Chaque action importante est enregistrée.

Horodatage

Chaque événement possède une date et une heure.

Responsabilité

Chaque action est associée à un utilisateur identifié.

Versionning

Chaque document possède un historique de versions.

Intégrité

Une preuve réglementaire ne peut pas être modifiée sans laisser de trace.

Transparence

Le cabinet doit être capable d'expliquer chaque recommandation.

6. Les preuves réglementaires

La plateforme distingue les documents des preuves.

Un document est un fichier.

Une preuve est un ensemble d'informations démontrant qu'une action a bien été réalisée.

Exemple :

Le devoir de conseil constitue un document.

La preuve comprend également :

la date de génération ;
l'utilisateur ;
les produits étudiés ;
les produits écartés ;
les raisons du choix ;
les validations ;
la signature.

La plateforme conserve l'ensemble de ces éléments.

7. La conformité continue

La conformité n'est pas contrôlée uniquement à la fin d'un Projet.

Elle est évaluée tout au long de son cycle de vie.

À chaque étape, la plateforme vérifie automatiquement :

les informations disponibles ;
les pièces manquantes ;
les validations nécessaires ;
les documents obligatoires ;
les signatures attendues.

Cette approche permet de détecter les anomalies dès leur apparition.

8. Les rôles dans la conformité

Chaque acteur possède un rôle.

Le Conseiller

Recueille les besoins.

Analyse.

Conseille.

Le Client

Fournit les informations.

Valide.

Signe.

Le Collaborateur

Contrôle.

Prépare.

Complète.

Le Responsable Conformité

Supervise.

Audite.

Contrôle.

Les Agents IA

Assistants.

Ils peuvent :

contrôler ;
vérifier ;
signaler ;
préparer.

Ils ne valident jamais une obligation réglementaire.

9. Les objectifs de la plateforme

La plateforme doit permettre au cabinet de démontrer à tout moment :

pourquoi une recommandation a été formulée ;
sur quelles informations elle repose ;
quels produits ont été étudiés ;
pourquoi certains produits ont été écartés ;
quels documents ont été remis ;
quand ils ont été remis ;
quand ils ont été signés.

L'ensemble de ces informations constitue le dossier réglementaire.

10. Critères de validation

La conformité sera considérée comme intégrée lorsque :

chaque Projet constitue un dossier réglementaire complet ;
toutes les actions importantes sont historisées ;
les preuves sont conservées ;
les documents sont versionnés ;
les utilisateurs sont identifiés ;
les Agents IA interviennent uniquement comme assistants ;
la plateforme est capable de démontrer le respect des obligations réglementaires à tout moment.



PARTIE 2 — Le recueil des besoins
11. Philosophie du recueil des besoins

Le recueil des besoins constitue la première étape du devoir de conseil.

Il ne s'agit pas d'un simple questionnaire.

Il représente la base de toutes les recommandations formulées par le cabinet.

Aucune proposition ne doit être réalisée sans un recueil des besoins adapté au Projet concerné.

Le recueil des besoins est intégré à la plateforme et ne repose jamais sur un outil externe visible par le client.

12. Un recueil des besoins par Projet

Chaque Projet possède son propre recueil des besoins.

Un Client peut donc disposer de plusieurs recueils différents.

Exemples :

Projet Assurance emprunteur
Projet Protection de la famille
Projet Assurance professionnelle
Projet Retraite
Projet Assurance vie

Chaque recueil est indépendant.

13. Les formulaires intelligents

Les formulaires sont dynamiques.

Ils s'adaptent automatiquement :

au type de Projet ;
au profil du Client ;
aux réponses déjà connues ;
aux obligations réglementaires.

La plateforme ne pose jamais deux fois la même question lorsque l'information est déjà disponible et toujours valable.

14. Organisation du questionnaire

Le questionnaire est structuré en sections.

Exemple générique :

Selon le type de Projet.

Situation professionnelle

Selon le type de Projet.

Documents nécessaires au traitement.

15. Questions conditionnelles

Certaines questions apparaissent uniquement lorsque cela est nécessaire.

Exemples :

Si le Client indique être chef d'entreprise :

→ affichage des questions professionnelles.

Si le Client indique avoir des enfants :

→ affichage des questions relatives à la protection familiale.

Si le Client possède déjà une assurance emprunteur :

→ affichage des questions relatives au contrat existant.

Le formulaire reste simple et pertinent.

16. Sauvegarde automatique

Le recueil des besoins est enregistré automatiquement.

Le Client peut interrompre sa saisie puis la reprendre ultérieurement.

Chaque modification importante est historisée.

17. Mise à jour des informations

Les informations permanentes sont enregistrées dans la fiche Client.

Les informations propres au besoin restent enregistrées dans le Projet.

Exemple :

Un changement d'adresse met à jour la fiche Client.

Le montant du prêt reste enregistré uniquement dans le Projet Assurance emprunteur.

Cette séparation évite les incohérences.

18. Contrôles automatiques

Pendant la saisie, la plateforme vérifie notamment :

les champs obligatoires ;
la cohérence des réponses ;
les pièces manquantes ;
les incohérences détectées ;
les informations déjà connues.

Les anomalies sont signalées immédiatement.

19. Exploitation du recueil des besoins

Une fois validé, le recueil des besoins alimente automatiquement :

le Projet ;
la fiche Client ;
les Workflows ;
le Domaine Produits ;
le Domaine Partenaires ;
le devoir de conseil ;
les Agents IA ;
Google Workspace pour la génération documentaire.

Aucune ressaisie n'est nécessaire.

20. Historique et versionning

Chaque version du recueil des besoins est conservée.

L'historique précise :

la date ;
l'auteur de la modification ;
les réponses modifiées ;
le motif (si applicable).

Le cabinet peut ainsi démontrer l'évolution des besoins du Client au fil du temps.

21. Critères de validation

Le recueil des besoins sera considéré comme conforme lorsque :

chaque Projet possède son propre recueil des besoins ;
les formulaires sont dynamiques et adaptés au type de Projet ;
les informations permanentes sont distinguées des informations propres au Projet ;
les contrôles de cohérence sont automatiques ;
les données alimentent directement les autres domaines de la plateforme ;
toutes les modifications sont historisées et versionnées.

Le conseiller ne construit pas ce dossier manuellement.

La plateforme le constitue automatiquement au fil de l'avancement du Projet.

L'objectif est de permettre au cabinet de démontrer, à tout moment, le respect de ses obligations réglementaires.

26. Score de conformité

29. Journal d'audit

les validations ;
les signatures ;
les interventions IA ;
les exécutions de Workflows.

Ce journal est inaltérable.

30. Archivage réglementaire

À la clôture d'un Projet :

Le dossier réglementaire est figé.

Les documents restent accessibles selon les droits.

Les obligations de conservation sont appliquées conformément aux règles en vigueur.

Aucun document ne peut être supprimé sans autorisation.

31. Export du dossier

La plateforme permet de générer un export complet du dossier réglementaire.

Cet export peut être utilisé notamment :

lors d'un contrôle ACPR ;
lors d'un audit interne ;
lors d'un litige ;
lors d'une demande du Client.

L'export comprend les documents, l'historique et le journal d'audit.

32. Critères de validation

Le dossier réglementaire sera considéré comme conforme lorsque :

chaque Projet dispose automatiquement de son dossier réglementaire ;
les documents obligatoires sont identifiés selon le type de Projet ;
les contrôles sont réalisés automatiquement ;
un score de conformité est calculé ;
les alertes sont générées en cas d'anomalie ;
toutes les dérogations sont historisées ;
le journal d'audit est complet et inaltérable ;
le dossier peut être exporté à tout moment.


PARTIE 4 — Le cycle de conformité
33. Philosophie

La conformité ne constitue pas une étape finale du Projet.

Elle accompagne le Projet depuis sa création jusqu'à son archivage.

Chaque étape importante du Projet fait l'objet d'un contrôle automatique.

L'objectif est de détecter les anomalies le plus tôt possible afin d'éviter toute non-conformité.

Chaque étape possède ses propres exigences réglementaires.

35. Contrôles automatiques

À chaque changement d'étape, la plateforme vérifie automatiquement :

les documents obligatoires ;
les signatures attendues ;
les informations manquantes ;
les validations nécessaires ;
les incohérences détectées.

Les contrôles sont réalisés sans intervention de l'utilisateur.

36. Les points de blocage

Certaines étapes peuvent empêcher la poursuite normale d'un Projet.

Exemples :

absence du recueil des besoins ;
devoir de conseil non validé ;
pièce justificative obligatoire manquante ;
signature indispensable absente.

Les règles de blocage sont paramétrables selon le type de Projet et la politique du cabinet.

Un administrateur peut autoriser une dérogation lorsque cela est justifié.

37. Les points d'alerte

D'autres situations ne bloquent pas le Projet mais nécessitent une attention particulière.

Exemples :

document arrivant prochainement à expiration ;
absence de réponse du client ;
délai de traitement dépassé ;
relance partenaire en attente.

Ces alertes sont affichées dans les tableaux de bord et sur la fiche Projet.

38. Les contrôles périodiques

La plateforme réalise également des contrôles réguliers.

Exemples :

vérification des pièces expirées ;
contrôle des consentements ;
suivi des échéances ;
contrôle des formations des mandataires ;
contrôle des conventions partenaires.

Ces contrôles peuvent être quotidiens, hebdomadaires ou mensuels.

39. Les contrôles des Agents IA

Les Agents IA assistent les utilisateurs en détectant automatiquement :

des pièces manquantes ;
des incohérences dans les réponses ;
des contradictions entre les documents ;
des oublis dans le devoir de conseil ;
des opportunités de mise à jour.

Ils ne prennent jamais de décision réglementaire.

Ils proposent.

L'utilisateur décide.

40. Les contrôles humains

Certaines validations restent exclusivement humaines.

Exemples :

validation finale du devoir de conseil ;
recommandation d'un produit ;
validation d'une dérogation ;
clôture d'un Projet.

La plateforme identifie clairement les validations réalisées par un utilisateur et celles proposées par un Agent IA.

41. Historique des contrôles

Chaque contrôle est enregistré.

Pour chaque contrôle, la plateforme conserve :

la date ;
le type de contrôle ;
le résultat ;
l'utilisateur ou l'Agent IA concerné ;
les anomalies détectées ;
les actions correctives réalisées.

Cet historique est intégré au dossier réglementaire.

42. Tableau de bord conformité

Les utilisateurs autorisés disposent d'un tableau de bord dédié.

Il permet notamment de suivre :

les Projets non conformes ;
les documents manquants ;
les signatures en attente ;
les contrôles à réaliser ;
les dérogations ouvertes ;
les alertes critiques ;
le score global de conformité du cabinet.
43. Critères de validation

Le cycle de conformité sera considéré comme conforme lorsque :

les contrôles sont effectués tout au long du Projet ;
les blocages et alertes sont clairement distingués ;
les contrôles périodiques sont automatisés ;
les Agents IA assistent les utilisateurs sans se substituer à eux ;
les validations humaines sont identifiées ;
l'ensemble des contrôles est historisé ;
les responsables disposent d'une vision claire de l'état de conformité du cabinet.

L'objectif est d'anticiper les risques plutôt que de les découvrir lors d'un contrôle.

les versions des documents ;
les validations ;
les signatures ;
les échanges ;

une évolution d'un formulaire ;
une évolution d'un Workflow ;
une évolution d'un Agent IA ;
une évolution documentaire ;
une évolution des procédures internes.

La conformité devient ainsi un levier d'amélioration continue.

Les Agents IA participent au pilotage de la conformité.

Ils peuvent notamment :

détecter des anomalies récurrentes ;
identifier des tendances ;
proposer des améliorations ;
préparer des rapports ;
assister les responsables conformité.

Ils ne prennent jamais de décision réglementaire.

Toutes leurs propositions sont soumises à validation humaine.

53. Critères de validation

le cabinet dispose d'une vision globale de son niveau de conformité ;
les indicateurs sont calculés automatiquement ;
les non-conformités sont tracées jusqu'à leur résolution ;
les audits peuvent être réalisés directement dans la plateforme ;
les contrôles ACPR peuvent être préparés rapidement ;
les Agents IA assistent le pilotage sans se substituer aux responsables ;
la plateforme favorise une démarche d'amélioration continue.


PARTIE 6 — Les principes fondamentaux de la conformité EJ Assurances
54. La conformité comme service

La conformité n'est pas une contrainte.

Elle constitue un service rendu :

au Client ;
au Conseiller ;
au Cabinet ;
aux Partenaires.

Elle garantit la qualité du conseil, la traçabilité des décisions et la protection de toutes les parties.

55. Une conformité intégrée

La conformité est présente dans l'ensemble de la plateforme.

Elle intervient notamment dans :

les formulaires ;
les Projets ;
les Documents ;
les Produits ;
les Partenaires ;
les Contrats ;
les Interactions ;
les Workflows ;
les Agents IA.

Aucun domaine ne fonctionne indépendamment de la conformité.

56. Une conformité démontrable

Le cabinet doit pouvoir démontrer à tout moment :

les besoins exprimés par le Client ;
les informations recueillies ;
les solutions étudiées ;
les produits écartés ;
les raisons des recommandations ;
les documents remis ;
les validations obtenues ;
les signatures réalisées.

La plateforme doit permettre de produire ces éléments sans reconstitution manuelle.

57. Une conformité proportionnée

Les exigences réglementaires varient selon le type de Projet.

La plateforme adapte automatiquement :

les formulaires ;
les documents attendus ;
les contrôles ;
les validations ;
les Workflows.

Chaque Projet applique uniquement les exigences qui lui sont applicables.

58. Une conformité collaborative

La conformité repose sur plusieurs acteurs.

Le Conseiller recueille les besoins.

Le Collaborateur prépare les dossiers.

Le Responsable conformité supervise.

Le Client fournit les informations et valide les documents.

Les Agents IA assistent les utilisateurs.

Chaque acteur intervient dans son périmètre de responsabilité.

59. Une conformité évolutive

Les exigences réglementaires évoluent.

La plateforme doit permettre :

d'ajouter de nouveaux contrôles ;
d'ajouter de nouveaux documents ;
d'ajouter de nouvelles règles ;
de modifier les procédures existantes.

Ces évolutions ne doivent pas remettre en cause les Projets déjà traités.

Les anciennes versions restent consultables.

60. Une conformité pilotée par les données

Les contrôles reposent sur les données enregistrées dans la plateforme.

La qualité des données conditionne la qualité de la conformité.

Les utilisateurs sont donc encouragés à compléter les informations au plus tôt dans le cycle de vie du Projet.

La plateforme contrôle automatiquement la cohérence de ces données.

61. Une conformité assistée par l'intelligence artificielle

Les Agents IA peuvent notamment :

analyser les dossiers ;
détecter des anomalies ;
signaler des incohérences ;
proposer des actions correctives ;
préparer des rapports ;
assister les audits.

Les Agents IA ne remplacent jamais le jugement professionnel ni les validations réglementaires.

Ils agissent comme des assistants spécialisés.

62. Les engagements d'EJ Assurances

La plateforme est conçue pour respecter les principes suivants :

protéger les intérêts du Client ;
garantir la qualité du conseil ;
assurer une traçabilité complète ;
limiter les risques de non-conformité ;
simplifier le travail des utilisateurs ;
favoriser l'amélioration continue ;
intégrer la conformité dans chaque étape du Projet.

Ces engagements guident toutes les évolutions de la plateforme.

63. Critères de validation du Domaine Conformité

Le Domaine Conformité sera considéré comme conforme lorsque :

chaque Projet dispose d'un dossier réglementaire complet ;
les obligations sont adaptées au type de Projet ;
les contrôles sont automatiques et historisés ;
les documents sont versionnés ;
les audits sont préparés directement depuis la plateforme ;
les non-conformités sont suivies jusqu'à leur résolution ;
les Agents IA interviennent uniquement dans un rôle d'assistance ;
les utilisateurs disposent d'une vision claire de l'état de conformité de leurs Projets.
64. Conclusion

Le Domaine Conformité constitue un pilier fondamental de la plateforme EJ Assurances.

Il ne vise pas uniquement à répondre aux obligations réglementaires.

Il contribue à améliorer la qualité du conseil, à sécuriser les utilisateurs et à renforcer la confiance des Clients.

En intégrant la conformité au cœur des processus métier, la plateforme permet au cabinet de se concentrer sur sa mission principale : accompagner et protéger durablement ses Clients.
