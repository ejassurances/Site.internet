PARTIE 1 — Philosophie et objectifs
1. Objet du document

Ce document définit l'architecture complète du site internet EJ Assurances.

Il constitue la référence unique pour :

les designers UX/UI ;
les développeurs ;
les rédacteurs de contenu ;
les futurs collaborateurs ;
les partenaires techniques.

L'objectif est que toute personne participant au projet sache exactement comment le site est organisé et pourquoi.

2. Philosophie générale

Le site internet EJ Assurances n'est pas un catalogue de produits.

Il est la porte d'entrée de tout l'écosystème EJ Assurances.

Sa mission est de :

rassurer ;
informer ;
démontrer notre expertise ;
accompagner ;
transformer un visiteur en client.

Le site doit toujours donner l'impression d'être accompagné par un cabinet humain et moderne.

Le visiteur ne doit jamais avoir l'impression d'être sur un comparateur d'assurances.

3. Les objectifs du site

Le site poursuit cinq objectifs.

Objectif n°1 : Générer des contacts qualifiés

Chaque page doit conduire naturellement vers :

une prise de rendez-vous ;
une demande de rappel ;
une simulation ;
ou une ouverture de projet.
Objectif n°2 : Démontrer notre expertise

Le visiteur doit comprendre immédiatement que le cabinet possède une expertise reconnue sur :

l'assurance emprunteur ;
la protection familiale ;
la coparentalité ;
les assurances professionnelles.
Objectif n°3 : Créer un climat de confiance

Le site doit montrer :

notre pédagogie ;
notre transparence ;
notre conformité ;
notre proximité.

Avant d'acheter une assurance, un client achète de la confiance.

Objectif n°4 : Alimenter le CRM

Chaque interaction importante doit enrichir automatiquement le CRM :

formulaire ;
demande de rendez-vous ;
téléchargement ;
dépôt de document ;
simulation.

Aucune saisie ne doit être perdue.

Objectif n°5 : Alimenter notre référencement

Le site doit devenir une référence :

Google ;
ChatGPT ;
Claude ;
Gemini ;
Perplexity ;
Mistral.

Chaque page doit pouvoir être comprise aussi bien par un humain que par une IA.

4. Les trois univers du site

Le projet est composé de trois univers.

Univers 1 : Site public

Accessible à tous.

Objectif :

Découvrir EJ Assurances.

Univers 2 : Espaces sécurisés

Accessible après connexion.

Il comprend :

espace client ;
espace mandataire ;
espace collaborateur ;
espace prescripteur.
Univers 3 : Plateforme métier

Invisible du public.

Elle comprend :

CRM ;
GED ;
conformité ;
partenaires ;
produits ;
IA ;
Google Workspace.
5. Principe fondamental

Le visiteur ne doit jamais voir la complexité.

Le système peut être extrêmement puissant.

L'expérience utilisateur doit rester extrêmement simple.

Le visiteur ne doit jamais avoir à comprendre :

le CRM ;
les workflows ;
les automatisations ;
Google Workspace ;
les IA.

Il doit simplement avoir le sentiment que tout est fluide.

6. Ce que le site ne doit jamais devenir

Le site ne doit jamais devenir :

un comparateur ;
un catalogue de garanties ;
une succession de formulaires ;
un site institutionnel froid ;
un jargon d'assureur.

Nous voulons raconter des parcours de vie.

Pas vendre des garanties.

PARTIE 3 — Les parcours utilisateurs

Objectif : définir précisément comment chaque type de visiteur découvre EJ Assurances, interagit avec le site, puis devient un utilisateur de la plateforme. Chaque parcours doit être fluide, rassurant et limiter les frictions.

19. Philosophie des parcours

Le site n'est pas construit autour de nos produits.

Il est construit autour des besoins des visiteurs.

Chaque parcours doit répondre à trois questions :

Pourquoi suis-je ici ?
Que dois-je faire maintenant ?
Que va-t-il se passer ensuite ?

À aucun moment le visiteur ne doit avoir l'impression d'être abandonné ou de ne plus savoir où il en est.

Étape 4 : Confirmation

Le visiteur reçoit immédiatement :

26. Les points de rupture

Le site doit identifier les moments où un visiteur peut abandonner.

Exemples :

formulaire trop long ;
manque de compréhension ;
attente trop importante ;
absence de réponse ;
demande de documents trop tôt.

Ces points devront être mesurés afin d'améliorer continuellement les parcours.

27. Les règles de conception des parcours

Tous les parcours doivent respecter les principes suivants :

un objectif clair par étape ;
une seule action principale ;
un langage simple ;
un minimum de saisie ;
une sauvegarde automatique lorsque c'est possible ;
aucune ressaisie d'information déjà connue.
28. Principe fondamental

Le visiteur ne remplit jamais un formulaire "pour le cabinet".

Il répond à des questions qui permettront au cabinet de mieux le protéger.

Cette différence de philosophie doit se retrouver dans tous les textes du site.

29. Critères de validation

Les parcours seront considérés comme validés lorsque :

chaque type de visiteur dispose d'un chemin clair ;
toute création de formulaire génère automatiquement un prospect dans le CRM ;
aucun parcours ne se termine sans proposition d'action ;
les espaces connectés prennent naturellement le relais après la création du prospect ;
le parcours reste compréhensible même pour une personne n'ayant aucune connaissance du monde de l'assurance.

PARTIE 4 — Les espaces connectés

Objectif : définir les différents espaces utilisateurs de la plateforme, leur rôle, leurs objectifs et leurs interactions. Le site public génère de la confiance ; les espaces connectés prolongent cette confiance par une expérience fluide, moderne et collaborative.

30. Philosophie des espaces connectés

Les espaces connectés ne sont pas des "extranets".

Ils constituent le prolongement naturel de la relation entre EJ Assurances et ses utilisateurs.

Chaque espace doit répondre aux mêmes principes :

simple à utiliser ;
rapide ;
sécurisé ;
moderne ;
centré sur les besoins de son utilisateur.

L'objectif est que chaque utilisateur retrouve uniquement les informations et fonctionnalités qui le concernent.

Espace Client

39. Continuité de l'expérience

Un client qui se connecte doit immédiatement retrouver les éléments évoqués lors de son parcours public :

ses documents ;
son conseiller ;
les prochaines étapes.

Il ne doit jamais avoir l'impression de changer d'univers.

40. Critères de validation

chaque rôle dispose d'un espace dédié ;
les droits sont clairement définis ;
le design est homogène ;
la navigation est cohérente ;
les informations sont contextualisées ;
chaque utilisateur retrouve rapidement les actions qui le concernent.


PARTIE 5 — Architecture fonctionnelle des échanges
41. Philosophie générale

L'utilisateur ne doit jamais avoir à se demander où se trouve une information.

Chaque donnée est enregistrée une seule fois.

Ensuite, elle est mise à disposition des différents modules selon les droits des utilisateurs.

Le principe fondamental de la plateforme est le suivant :

Une donnée → Une source de vérité → Plusieurs vues.

Aucune duplication inutile n'est autorisée.

42. Le Projet : le cœur de la plateforme

Contrairement aux CRM classiques qui placent le client au centre, EJ Assurances place le Projet au cœur de la plateforme.

Un projet représente un besoin exprimé par un client.

Exemples :

Assurance emprunteur
Protection familiale
Assurance professionnelle
Retraite
Assurance vie
Prévoyance

Chaque projet regroupe l'ensemble des informations nécessaires à son traitement.

Un client peut avoir plusieurs projets.

Chaque projet possède son propre historique, ses documents, ses échanges, ses tâches et ses décisions.

43. Le parcours d'une donnée

Exemple :

Le client remplit un formulaire.

↓

Le CRM crée automatiquement :

le prospect (si inexistant) ;
le projet ;
la première interaction.

↓

Les informations sont enregistrées dans la base de données.

↓

Les différents modules récupèrent uniquement les informations dont ils ont besoin.

Par exemple :

la GED reçoit les documents ;
le calendrier reçoit les rendez-vous ;
le module conformité prépare le recueil des besoins ;
les agents IA analysent le contexte ;
Google Workspace prépare les documents.

Le client n'a jamais besoin de ressaisir les mêmes informations.

44. Les modules communiquent entre eux

Tous les modules de la plateforme doivent être interconnectés.

Exemple :

45. Google Workspace : un moteur documentaire invisible

Google Workspace n'est pas utilisé comme interface utilisateur.

Il constitue le moteur documentaire de la plateforme.

Le client ne voit jamais :

Google Drive ;
Google Docs ;
Google Sheets.

Il interagit uniquement avec l'interface EJ Assurances.

En arrière-plan, le système peut :

créer un dossier Drive ;
générer un document Google Docs ;
alimenter un Google Sheet ;
envoyer une demande de signature ;
récupérer le document signé.

Le CRM reste toujours la référence.

Google Workspace est un outil d'exécution.

46. La bibliothèque documentaire

Tous les documents de la plateforme proviennent d'une bibliothèque documentaire unique.

Un document est stocké une seule fois.

Puis il peut être utilisé dans plusieurs modules.

Exemple :

Un IPID peut être visible :

dans la fiche produit ;
dans le devoir de conseil ;
dans l'espace client ;
dans le projet ;
dans l'espace partenaire ;
dans le module conformité.

Le document n'est jamais dupliqué.

Ils ne modifient jamais un projet sans traçabilité.

Toutes leurs actions sont historisées.

51. Le principe du 360°

Tous les utilisateurs travaillent sur le même projet.

Chaque utilisateur dispose d'une vue adaptée à son rôle.

L'administrateur voit :

les indicateurs ;
les statistiques.

Les IA voient uniquement les informations nécessaires à leur mission.

Le projet reste unique.

Seule la vue change.

52. Critères de validation

L'architecture fonctionnelle sera considérée comme conforme lorsque :

chaque donnée possède une source unique ;
chaque module communique avec les autres ;
Google Workspace reste invisible pour l'utilisateur ;
les partenaires alimentent le catalogue produits ;
le catalogue produits alimente le devoir de conseil ;
le devoir de conseil alimente le projet ;
les IA travaillent uniquement sur le contexte du projet ;
chaque acteur dispose d'une vision adaptée de ce même projet.
