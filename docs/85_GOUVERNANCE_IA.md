PARTIE 1 — Philosophie de la gouvernance IA
1. Objet du document

Ce document définit les règles de gouvernance applicables à l'ensemble des Services IA de la plateforme EJ Assurances.

Il décrit notamment :

les principes de fonctionnement ;
les responsabilités ;
les règles de validation ;
les évolutions ;
les contrôles ;
les audits ;
la supervision.

Il constitue la référence officielle de gouvernance des Services IA.

2. Philosophie générale

L'intelligence artificielle constitue un outil d'assistance.

Elle ne constitue jamais une autorité décisionnelle.

Les Services IA interviennent pour :

assister ;
préparer ;
expliquer ;
organiser ;
contrôler ;
proposer.

Les décisions engageant le cabinet restent exclusivement humaines.

3. Les principes fondateurs

Tous les Services IA respectent les principes suivants :

spécialisation métier ;
transparence ;
explicabilité ;
traçabilité ;
sécurité ;
conformité ;
amélioration continue.

Aucun Service IA ne peut s'affranchir de ces principes.

4. Les responsabilités

La responsabilité est répartie entre plusieurs acteurs.

Le cabinet

Définit les règles métier.

Les responsables métier

Valident les connaissances.

Les administrateurs

Paramètrent les Services IA.

Les utilisateurs

Décident d'accepter ou non les propositions.

Les Services IA

Préparent les analyses et exécutent les actions autorisées.

Chaque acteur conserve son périmètre de responsabilité.

5. Le principe de responsabilité humaine

Toute décision ayant une conséquence :

juridique ;
réglementaire ;
contractuelle ;
financière ;
commerciale stratégique ;

doit être validée par une personne habilitée.

Aucun Service IA ne peut se substituer à cette responsabilité.

6. Les connaissances validées

Les Services IA utilisent exclusivement :

des documents validés ;
des procédures validées ;
des règles métier validées ;
des bases de connaissances approuvées.

Ils ne créent jamais eux-mêmes une nouvelle règle métier.

7. Les objectifs de la gouvernance

La gouvernance vise à garantir :

la qualité des réponses ;
la conformité réglementaire ;
la confiance des utilisateurs ;
la sécurité des données ;
la stabilité des Services IA ;
l'amélioration continue.
8. Critères de validation

La gouvernance sera considérée comme conforme lorsque :

les responsabilités sont clairement définies ;
les décisions restent humaines ;
les connaissances sont validées avant utilisation ;
les Services IA respectent les principes fondateurs ;
les utilisateurs conservent la maîtrise des décisions ;
les évolutions sont contrôlées.

PARTIE 2 — Cycle de vie d'un Service IA
9. Philosophie

Un Service IA est un composant vivant de la plateforme.

Il possède un cycle de vie complet.

Comme un collaborateur, il est :

conçu ;
testé ;
mis en production ;
supervisé ;
amélioré ;
éventuellement remplacé.

La gouvernance garantit que ces évolutions restent maîtrisées.

10. Création d'un nouveau Service IA

Aucun nouveau Service IA ne peut être créé sans respecter le processus suivant :

Définition du besoin métier.
Définition de la mission.
Définition des responsabilités.
Définition des limites.
Définition des connaissances.
Définition des droits d'accès.
Définition des Workflows concernés.
Validation par le référentiel documentaire.

Le développement ne peut commencer qu'après validation de ces étapes.

11. Développement

Le développement d'un Service IA respecte les principes suivants :

séparation entre métier et technique ;
utilisation des objets métier existants ;
respect de l'architecture globale ;
documentation systématique ;
développement modulaire.

Le code doit toujours refléter le référentiel documentaire.

12. Phase de test

Avant toute mise en production, un Service IA est testé selon plusieurs niveaux.

Tests techniques
fonctionnement ;
performances ;
sécurité.
Tests métier
cohérence des réponses ;
respect des procédures ;
qualité des analyses.
Tests réglementaires
conformité ACPR ;
conformité DDA ;
conformité RGPD.
Tests utilisateurs
compréhension ;
ergonomie ;
pertinence.

Aucun Service IA n'est déployé sans validation de ces tests.

13. Mise en production

La mise en production d'un Service IA est une étape contrôlée.

Elle comprend :

validation documentaire ;
validation métier ;
validation technique ;
validation de sécurité ;
validation réglementaire.

Chaque mise en production est historisée.

14. Supervision continue

Une fois en production, chaque Service IA est supervisé en permanence.

Les éléments suivants sont suivis :

disponibilité ;
performances ;
erreurs ;
qualité des réponses ;
taux d'utilisation ;
retours utilisateurs.

Les incidents sont détectés automatiquement.

15. Amélioration continue

Chaque Service IA fait l'objet d'une amélioration continue.

Les évolutions peuvent provenir :

des utilisateurs ;
des collaborateurs ;
des responsables métier ;
des évolutions réglementaires ;
des nouveaux Produits ;
des nouveaux Partenaires.

Toute évolution suit le même processus de validation que lors de la création.

16. Gestion des versions

Chaque Service IA possède une version.

Toute évolution importante entraîne :

une nouvelle version ;
une documentation mise à jour ;
une historisation des modifications.

Les anciennes versions restent consultables à des fins d'audit.

17. Remplacement d'un Service IA

Un Service IA peut être remplacé.

Exemples :

changement de modèle d'IA ;
nouvelle technologie ;
nouvelle architecture.

Le remplacement ne doit jamais modifier :

son rôle métier ;
ses responsabilités ;
ses interfaces avec les autres Services.

Cette règle garantit la stabilité de la plateforme.

18. Désactivation

Un Service IA peut être désactivé temporairement.

Dans ce cas :

les Workflows concernés sont suspendus ou adaptés ;
les utilisateurs sont informés ;
les autres Services continuent à fonctionner lorsque cela est possible ;
toutes les actions sont historisées.

Aucune désactivation ne doit entraîner une perte de données.

19. Fin de vie

Lorsqu'un Service IA est définitivement retiré :

sa documentation est conservée ;
son historique est archivé ;
ses connaissances restent disponibles si elles sont encore pertinentes ;
les Workflows sont mis à jour.

La suppression définitive d'un Service IA reste exceptionnelle.

20. Critères de validation

Le cycle de vie d'un Service IA sera considéré comme conforme lorsque :

chaque Service suit un processus documenté de création ;
les phases de test sont obligatoires ;
les mises en production sont validées ;
les versions sont historisées ;
les évolutions sont maîtrisées ;
les remplacements sont transparents pour l'architecture ;
la désactivation et la fin de vie sont gérées sans perte de données.

PARTIE 3 — Gouvernance des connaissances
21. Philosophie

La qualité des Services IA dépend directement de la qualité de leurs connaissances.

Le cabinet applique un principe fondamental :

Une IA n'est jamais plus fiable que les connaissances qui lui sont fournies.

Les connaissances constituent donc un patrimoine stratégique du cabinet.

Elles sont gérées avec le même niveau d'exigence que les données métier.

22. Sources autorisées

Les Services IA utilisent uniquement des connaissances provenant de sources identifiées et validées.

Exemples :

Référentiel documentaire EJ Assurances ;
procédures internes ;
fiches Produits ;
fiches Partenaires ;
modèles documentaires ;
documentation réglementaire ;
conventions partenaires ;
base documentaire Google Drive.

Toute source doit être référencée.

23. Sources interdites

Les Services IA ne peuvent jamais construire leurs réponses à partir :

d'une conversation utilisateur non validée ;
d'une supposition ;
d'une expérience isolée ;
d'une information non vérifiée ;
d'un document obsolète ;
d'une source non identifiée.

Une information non validée ne devient jamais une connaissance officielle.

24. Cycle de validation d'une connaissance

Toute nouvelle connaissance suit le processus suivant :

Identification

↓

Analyse

↓

Validation métier

↓

Intégration dans la base de connaissances

↓

Versionnement

↓

Mise à disposition des Services IA

Aucune étape ne peut être ignorée.

25. Les retours d'expérience

Les collaborateurs peuvent proposer de nouveaux retours d'expérience.

Exemples :

nouvelle procédure partenaire ;
évolution d'un Produit ;
amélioration d'un Workflow ;
retour après un contrôle ACPR ;
bonne pratique observée.

Ces retours sont enregistrés comme propositions.

Ils ne deviennent exploitables qu'après validation.

26. Les responsables de validation

Chaque domaine possède un responsable.

Par exemple :

Produits → Responsable Produits ;
Partenaires → Responsable Réseau ;
Conformité → Responsable Conformité ;
Documentation → Responsable Documentation ;
CRM → Responsable Métier.

Le responsable valide ou refuse l'intégration d'une nouvelle connaissance.

27. Versionnement des connaissances

Toute connaissance possède :

une date de création ;
une version ;
un auteur ;
un validateur ;
une date de validation ;
un historique des modifications.

Les anciennes versions restent consultables.

28. Péremption

Certaines connaissances deviennent obsolètes.

Exemples :

évolution d'un Produit ;
changement réglementaire ;
nouvelle convention partenaire.

La plateforme peut marquer une connaissance comme :

Active ;
À vérifier ;
Obsolète ;
Archivée.

Les Services IA n'utilisent jamais une connaissance obsolète.

29. Audit des connaissances

Le cabinet peut réaliser un audit de la base de connaissances.

L'objectif est de vérifier :

la cohérence ;
l'actualité ;
la qualité ;
la complétude ;
la conformité.

Les résultats de ces audits permettent d'améliorer progressivement les Services IA.

30. Utilisation des connaissances

Lorsqu'un Service IA produit une réponse importante, il doit pouvoir indiquer :

quelles connaissances ont été utilisées ;
quelle version de ces connaissances a été consultée ;
à quelle date elles ont été validées.

Cette explicabilité renforce la confiance des utilisateurs.

31. Base de connaissances unique

Le cabinet maintient une base de connaissances unique.

Tous les Services IA consultent cette base.

Ils ne possèdent pas de copies indépendantes.

Cette règle garantit la cohérence des réponses dans toute la plateforme.

32. Critères de validation

La gouvernance des connaissances sera considérée comme conforme lorsque :

toutes les connaissances proviennent de sources identifiées ;
les nouvelles connaissances sont validées avant utilisation ;
les retours d'expérience suivent un processus de validation ;
chaque connaissance est versionnée ;
les connaissances obsolètes sont retirées de l'utilisation ;
les réponses importantes restent explicables ;
tous les Services IA utilisent une base de connaissances unique.

PARTIE 4 — Supervision, audit et amélioration continue
33. Philosophie

La gouvernance des Services IA ne s'arrête pas à leur mise en production.

Chaque Service IA fait l'objet d'une supervision permanente afin de garantir :

sa fiabilité ;
sa conformité ;
sa performance ;
sa sécurité ;
sa pertinence.

L'objectif n'est pas de surveiller les Services IA, mais de garantir qu'ils continuent à apporter de la valeur au cabinet.

34. Supervision en temps réel

Chaque Service IA est supervisé en continu.

Les indicateurs suivants sont notamment suivis :

disponibilité ;
temps de réponse ;
nombre d'interventions ;
taux de réussite ;
erreurs détectées ;
interruptions ;
alertes.

Ces informations alimentent le Cockpit de supervision.

35. Audit des décisions IA

Toutes les décisions préparées par un Service IA doivent pouvoir être auditées.

Pour chaque intervention importante, la plateforme conserve :

le Service IA concerné ;
le Projet concerné ;
la demande initiale ;
les connaissances utilisées ;
les règles appliquées ;
la réponse produite ;
la validation ou le refus de l'utilisateur.

Cet historique garantit l'explicabilité des interventions.

36. Indicateurs de qualité

La qualité d'un Service IA est évaluée selon plusieurs critères.

Exemples :

Qualité des réponses
taux d'acceptation ;
taux de modification ;
taux de rejet.
Qualité métier
conformité des propositions ;
cohérence avec les procédures ;
pertinence des recommandations.
Qualité technique
disponibilité ;
rapidité ;
stabilité.

Ces indicateurs permettent d'identifier les axes d'amélioration.

37. Gestion des incidents

Lorsqu'un incident est détecté :

il est enregistré ;
il est qualifié ;
son impact est évalué ;
les utilisateurs concernés sont identifiés ;
une action corrective est proposée.

Les incidents sont historisés afin d'éviter leur répétition.

38. Amélioration continue

L'amélioration des Services IA repose sur plusieurs sources :

retours des utilisateurs ;
retours des collaborateurs ;
audits internes ;
évolutions réglementaires ;
évolution des Produits ;
évolution des Partenaires ;
nouveaux besoins métier.

Toute amélioration suit le processus de gouvernance défini précédemment.

39. Gestion des anomalies

Une anomalie peut concerner :

une réponse incorrecte ;
une connaissance obsolète ;
une erreur documentaire ;
une incohérence métier ;
une erreur technique ;
une erreur de Workflow.

Chaque anomalie reçoit :

un niveau de criticité ;
un responsable ;
une date de traitement ;
un statut.
40. Évaluation périodique

Chaque Service IA fait l'objet d'une revue périodique.

Cette revue vérifie notamment :

la pertinence de sa mission ;
l'actualité de ses connaissances ;
la qualité de ses résultats ;
le respect de ses limites ;
son utilisation réelle ;
son impact sur le cabinet.

Cette revue peut conduire à :

une amélioration ;
une évolution ;
une fusion avec un autre Service ;
une mise hors service.
41. Tableau de bord de gouvernance

Le Responsable IA dispose d'un tableau de bord consolidé.

Il retrouve notamment :

Activité
interventions ;
utilisateurs ;
temps économisé.
Qualité
suggestions acceptées ;
suggestions rejetées ;
erreurs.
Connaissances
nouvelles connaissances ;
connaissances à valider ;
connaissances obsolètes.
Incidents
incidents ouverts ;
incidents critiques ;
incidents résolus.
Évolutions
nouvelles versions ;
Services en test ;
Services en production.
42. Critères de validation

La supervision des Services IA sera considérée comme conforme lorsque :

chaque Service est supervisé en continu ;
toutes les interventions importantes sont auditables ;
les indicateurs de qualité sont suivis ;
les incidents sont tracés et traités ;
les anomalies sont classifiées ;
chaque Service fait l'objet d'une revue périodique ;
le Responsable IA dispose d'une vision consolidée de l'ensemble des Services.
43. Principe d'amélioration permanente

La gouvernance des Services IA repose sur un principe simple :

Un Service IA n'est jamais considéré comme terminé.

Chaque retour utilisateur, chaque évolution réglementaire, chaque nouveau Produit, chaque nouveau Partenaire et chaque amélioration métier constitue une opportunité d'améliorer les Services.

L'objectif est de faire évoluer progressivement l'ensemble de l'écosystème IA tout en conservant une architecture stable, documentée et maîtrisée.

PARTIE 5 — Principes fondateurs de la gouvernance IA
44. Philosophie

La gouvernance des Services IA d'EJ Assurances repose sur une conviction simple :

L'intelligence artificielle est un accélérateur du métier, jamais un substitut au métier.

Les Services IA sont conçus pour assister les collaborateurs, sécuriser les processus et améliorer la qualité du conseil.

Ils ne remplacent ni l'expérience, ni le discernement, ni la responsabilité des professionnels.

45. Les dix principes fondateurs

L'ensemble des Services IA respecte les principes suivants.

1. La responsabilité reste humaine

Toute décision engageant le cabinet demeure sous la responsabilité d'une personne habilitée.

2. La donnée est unique

Les Services IA utilisent les objets métier de la plateforme.

Ils ne créent jamais de copies indépendantes.

3. Les connaissances sont validées

Aucune connaissance n'est utilisée sans validation préalable.

Les Services IA n'inventent jamais de règle métier.

4. Les actions sont explicables

Chaque proposition importante doit pouvoir être justifiée.

Les utilisateurs doivent comprendre le raisonnement suivi.

5. Les interventions sont historisées

Toutes les actions importantes sont enregistrées.

L'historique permet les audits, les contrôles et l'amélioration continue.

6. Les Services sont spécialisés

Chaque Service intervient uniquement dans son domaine de compétence.

La coopération remplace la polyvalence excessive.

7. Les Workflows orchestrent les Services

Les Services IA n'organisent pas eux-mêmes leurs interventions.

Ils exécutent les Workflows définis par la plateforme.

8. Les outils sont remplaçables

Les Services utilisent des connecteurs vers les outils externes.

Le métier reste indépendant des technologies utilisées.

9. L'amélioration est continue

Les Services évoluent progressivement grâce :

aux retours utilisateurs ;
aux évolutions réglementaires ;
aux nouvelles connaissances ;
aux améliorations techniques.
10. La confiance est prioritaire

Toute évolution doit renforcer la confiance :

des Clients ;
des Collaborateurs ;
des Partenaires ;
des autorités de contrôle.

La confiance prime toujours sur l'automatisation.

46. La place de l'humain

Le cabinet reste organisé autour des collaborateurs.

Les Services IA prennent en charge :

les tâches répétitives ;
les recherches ;
les préparations ;
les contrôles ;
les synthèses.

Les collaborateurs conservent :

la relation humaine ;
l'écoute ;
l'analyse ;
la décision ;
la responsabilité.

Cette complémentarité constitue un principe fondamental de la plateforme.

47. Vision stratégique

L'objectif d'EJ Assurances n'est pas de créer un cabinet automatisé.

L'objectif est de créer un cabinet augmenté.

L'intelligence artificielle permet aux équipes de consacrer davantage de temps :

au conseil ;
à la relation Client ;
à l'accompagnement ;
à la stratégie ;
à la création de valeur.

Elle réduit la charge administrative sans diminuer la qualité du travail humain.

48. Vision à long terme

La gouvernance IA est conçue pour accompagner durablement le développement du cabinet.

Elle doit permettre :

l'intégration de nouveaux Services IA ;
l'évolution des modèles d'intelligence artificielle ;
l'arrivée de nouvelles réglementations ;
l'ouverture à de nouveaux métiers (CGP, Crédit, CIF, Immobilier, etc.) ;
l'évolution des technologies documentaires et des partenaires.

L'architecture de gouvernance doit rester stable malgré les évolutions technologiques.

49. Engagement d'EJ Assurances

EJ Assurances s'engage à développer une intelligence artificielle :

responsable ;
explicable ;
sécurisée ;
documentée ;
conforme ;
utile ;
au service des utilisateurs.

La technologie n'est jamais une finalité.

Elle constitue un moyen d'améliorer durablement la qualité du conseil et l'efficacité opérationnelle.

50. Critères de validation

La gouvernance IA sera considérée comme conforme lorsque :

les principes fondateurs sont respectés par tous les Services ;
les responsabilités humaines sont clairement définies ;
les connaissances sont maîtrisées ;
les Services restent spécialisés ;
les outils externes restent découplés de l'architecture métier ;
la confiance des utilisateurs est préservée ;
l'amélioration continue est intégrée au fonctionnement de la plateforme.
51. Conclusion

La gouvernance IA constitue le cadre de référence de l'ensemble des Services IA de la plateforme EJ Assurances.

Elle garantit que l'intelligence artificielle reste un outil maîtrisé, transparent et conforme, au service des collaborateurs et des Clients.

En combinant une architecture métier robuste, une gouvernance rigoureuse et une amélioration continue, EJ Assurances construit une plateforme capable d'évoluer pendant de nombreuses années sans remettre en cause ses principes fondateurs.
