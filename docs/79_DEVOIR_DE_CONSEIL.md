PARTIE 1 — Philosophie du devoir de conseil
1. Objet du document

Ce document définit le fonctionnement du devoir de conseil au sein de la plateforme EJ Assurances.

Il constitue la référence métier pour :

la préparation des recommandations ;
la justification des choix proposés au Client ;
la traçabilité des analyses ;
la conformité réglementaire ;
l'exploitation des Produits et des Partenaires.

Toutes les recommandations formulées par le cabinet devront respecter les principes définis dans ce document.

2. Philosophie générale

Le devoir de conseil n'est pas un document.

Le devoir de conseil est un processus.

Le document remis au Client n'est que la synthèse de ce processus.

La plateforme doit être capable de démontrer :

comment les besoins ont été identifiés ;
quelles solutions ont été étudiées ;
pourquoi certaines solutions ont été retenues ;
pourquoi d'autres ont été écartées.

Le document final constitue la conséquence logique de l'ensemble du Projet.

3. Le Projet comme origine du devoir de conseil

Chaque devoir de conseil est rattaché à un Projet.

Un Projet peut comporter plusieurs versions du devoir de conseil.

Exemples :

première étude ;
nouvelle proposition ;
modification après échange avec le Client ;
évolution du besoin.

Chaque version est conservée.

Aucune version n'est supprimée.

4. Les sources d'information

Le devoir de conseil est construit automatiquement à partir des informations disponibles dans la plateforme.

Il exploite notamment :

le recueil des besoins ;
la fiche Client ;
les Produits étudiés ;
les Partenaires ;
les Documents réglementaires ;
les échanges ;
les analyses du conseiller.

Aucune ressaisie n'est nécessaire.

5. Les objectifs

Le devoir de conseil poursuit plusieurs objectifs.

Comprendre

Le besoin réel du Client.

Comparer

Les différentes solutions envisageables.

Justifier

Les recommandations proposées.

Informer

Le Client de manière claire et compréhensible.

Prouver

Le respect des obligations réglementaires.

6. Les principes fondamentaux

Le devoir de conseil repose sur plusieurs principes.

Individualisation

Chaque recommandation est personnalisée.

Objectivité

Les solutions sont analysées selon les besoins du Client.

Transparence

Les critères de sélection sont explicités.

Traçabilité

Toutes les étapes sont historisées.

Évolutivité

Le devoir de conseil peut être mis à jour en cas d'évolution du Projet.

7. Les acteurs

Plusieurs acteurs interviennent dans la préparation du devoir de conseil.

Le Conseiller

Analyse les besoins.

Sélectionne les solutions.

Valide les recommandations.

Le Client

Exprime ses besoins.

Fournit les informations.

Valide les choix.

Les Partenaires

Fournissent les Produits et leur documentation.

Les Agents IA

Assistent le conseiller.

Ils peuvent :

préparer une comparaison ;
détecter des incohérences ;
proposer des arguments ;
vérifier la cohérence avec le recueil des besoins.

Ils ne choisissent jamais le Produit à la place du conseiller.

8. Le rôle de la plateforme

La plateforme automatise :

la préparation des analyses ;
la génération documentaire ;
la récupération des informations ;
la comparaison des Produits ;
la traçabilité ;
l'historisation.

Le conseiller reste responsable du contenu final.

9. Les preuves conservées

La plateforme conserve notamment :

le recueil des besoins ;
les Produits étudiés ;
les Produits écartés ;
les critères de comparaison ;
les documents remis ;
les validations ;
les signatures ;
les versions successives.

Ces éléments constituent les preuves du devoir de conseil.

10. Critères de validation

Le devoir de conseil sera considéré comme conforme lorsque :

il est rattaché à un Projet ;
il est construit à partir des données du Projet ;
les recommandations sont justifiées ;
les solutions étudiées sont tracées ;
les versions sont historisées ;
les Agents IA interviennent uniquement comme assistants ;
le conseiller valide systématiquement la version finale.



PARTIE 2 — La construction du devoir de conseil
11. Philosophie

Le devoir de conseil ne se rédige pas.

Il se construit progressivement tout au long du Projet.

Chaque information recueillie enrichit automatiquement le futur document.

À aucun moment le conseiller ne doit repartir d'une page blanche.

La plateforme prépare le travail, le conseiller exerce son expertise.

14. Identification des besoins

La plateforme ne se limite pas au besoin exprimé.

Elle distingue :

Le besoin exprimé

Ce que demande le Client.

Le besoin identifié

Ce que le conseiller déduit de son analyse.

Les besoins complémentaires

Opportunités détectées au cours de l'étude.

Exemples :

protection du conjoint ;
prévoyance ;
garantie perte d'emploi ;
responsabilité du dirigeant ;
protection juridique.

Ces besoins complémentaires ne donnent jamais lieu à une vente forcée.

Ils permettent simplement au conseiller d'ouvrir une discussion pertinente.

15. Les solutions étudiées

Toutes les solutions réellement étudiées doivent être enregistrées.

Pour chaque solution, la plateforme conserve :

le Produit ;
le Partenaire ;
la date d'étude ;
les avantages ;
les limites ;
les observations du conseiller.

Cette traçabilité est essentielle pour démontrer la qualité de l'analyse.

16. Les solutions écartées

La plateforme doit également conserver les solutions non retenues.

Pour chaque solution écartée, un ou plusieurs motifs peuvent être indiqués.

Exemples :

garanties insuffisantes ;
exclusions importantes ;
coût inadapté ;
critères d'éligibilité non remplis ;
souhait du Client ;
meilleure solution identifiée.

Cette partie est importante en cas de contrôle ou de contestation.

17. Les critères de recommandation

La plateforme aide le conseiller à formaliser les critères ayant conduit à sa recommandation.

Exemples :

adéquation avec les besoins ;
niveau de garantie ;
rapport qualité/prix ;
qualité du service de gestion ;
rapidité de souscription ;
expérience avec le partenaire ;
exigences particulières du Client.

Ces critères sont personnalisables.

18. Intervention des Agents IA

Les Agents IA peuvent assister le conseiller en :

préparant un comparatif ;
détectant des incohérences ;
signalant des garanties manquantes ;
proposant une première rédaction.

Le conseiller reste responsable de la recommandation finale.

Chaque intervention IA est enregistrée dans l'historique.

19. Prévisualisation

Avant validation, la plateforme génère une prévisualisation du devoir de conseil.

Le conseiller peut :

modifier les commentaires ;
compléter les justifications ;
retirer une recommandation ;
ajouter une observation.

Le document n'est figé qu'après validation.

20. Critères de validation

La construction du devoir de conseil sera considérée comme conforme lorsque :

elle est alimentée automatiquement par les données du Projet ;
toutes les solutions étudiées sont tracées ;
les solutions écartées sont justifiées ;
les critères de recommandation sont explicités ;
les Agents IA assistent sans décider ;
le conseiller valide systématiquement la version présentée au Client.

PARTIE 3 — La comparaison des solutions et la recommandation
21. Philosophie

Le devoir de conseil ne consiste pas à présenter un seul produit.

Il consiste à démontrer que plusieurs solutions ont été analysées avant d'aboutir à une recommandation.

La plateforme doit donc permettre de comparer objectivement les solutions étudiées et de conserver la justification du choix final.

L'objectif n'est pas de retenir systématiquement le produit le moins cher ou le plus complet.

L'objectif est de retenir la solution la plus adaptée aux besoins du Client.

22. Constitution de la liste des solutions

Les solutions étudiées peuvent provenir de plusieurs sources :

Produits référencés dans le catalogue du cabinet ;
API des partenaires ;
Simulations réalisées sur les extranets ;
Études manuelles du conseiller ;
Recommandations proposées par les Agents IA.

Chaque solution est enregistrée dans le Projet.

23. Les critères de comparaison

La plateforme permet de comparer les solutions selon plusieurs axes.

Garanties
garanties incluses ;
garanties optionnelles ;
plafonds ;
franchises ;
exclusions.
Conditions
conditions d'accès ;
formalités médicales ;
délais ;
modalités de résiliation.
Aspect financier
cotisation ;
coût total estimé ;
frais éventuels ;
mode de paiement.
Qualité de service
qualité de gestion ;
délais de traitement ;
outils numériques ;
accompagnement proposé.
Adéquation

Chaque solution reçoit une appréciation sur son adéquation avec les besoins exprimés.

Cette appréciation n'est pas une note automatique.

Elle résulte de l'analyse du conseiller.

24. Les critères de décision

La plateforme ne classe pas automatiquement les produits.

Elle fournit les informations permettant au conseiller de prendre une décision éclairée.

Le conseiller indique ensuite les critères ayant motivé son choix.

Exemples :

meilleure couverture ;
meilleure adéquation avec le besoin ;
contraintes médicales ;
contraintes professionnelles ;
souhait exprimé par le Client ;
rapidité de mise en place ;
rapport garanties / coût.
25. Les solutions retenues

Une ou plusieurs solutions peuvent être retenues.

Pour chacune d'elles, la plateforme enregistre :

le Produit ;
le Partenaire ;
les raisons du choix ;
les avantages pour le Client ;
les éventuels points de vigilance.

Ces informations alimentent automatiquement le devoir de conseil.

26. Les solutions non retenues

Les solutions écartées restent visibles dans le Projet.

Pour chacune, la plateforme permet d'indiquer un ou plusieurs motifs.

Exemples :

garanties insuffisantes ;
exclusions incompatibles ;
tarif inadapté ;
critères d'éligibilité non remplis ;
absence de certaines garanties ;
préférence exprimée par le Client.

Cette traçabilité permet de démontrer que plusieurs solutions ont réellement été étudiées.

27. Recommandation finale

La recommandation finale est rédigée par le conseiller.

La plateforme prépare automatiquement une proposition à partir des données disponibles.

Le conseiller peut :

modifier le texte ;
compléter les arguments ;
supprimer des éléments ;
ajouter des observations.

La validation finale lui appartient exclusivement.

28. Présentation au Client

La version remise au Client doit être :

claire ;
compréhensible ;
personnalisée ;
exempte de jargon inutile.

Elle doit expliquer :

les besoins identifiés ;
les solutions étudiées ;
la solution recommandée ;
les raisons de cette recommandation ;
les principaux avantages ;
les principales limites.

L'objectif est que le Client comprenne le raisonnement suivi.

29. Signature et validation

Une fois présenté au Client :

le devoir de conseil est signé électroniquement ou validé selon le processus retenu par le cabinet ;
la version signée est figée ;
toute modification ultérieure entraîne la création d'une nouvelle version.

La version signée reste archivée dans le dossier réglementaire.

30. Critères de validation

Le processus de comparaison sera considéré comme conforme lorsque :

plusieurs solutions peuvent être étudiées ;
les critères de comparaison sont personnalisables ;
les motifs de choix et de rejet sont conservés ;
la recommandation est rédigée et validée par le conseiller ;
la version remise au Client est archivée et versionnée ;
le raisonnement ayant conduit à la recommandation peut être démontré à tout moment.

PARTIE 4 — La génération documentaire et le cycle de vie
31. Philosophie

Le devoir de conseil n'est jamais rédigé manuellement à partir d'un document vierge.

Il est généré automatiquement à partir des informations du Projet.

Le conseiller ne complète que les éléments nécessitant une expertise ou une justification personnalisée.

Cette approche garantit :

une homogénéité des documents ;
une réduction des erreurs ;
une meilleure traçabilité ;
un gain de temps important.
32. Génération automatique

Lorsque les étapes préalables sont validées, la plateforme génère automatiquement un projet de devoir de conseil.

La génération utilise notamment :

le recueil des besoins ;
les informations de la fiche Client ;
les Produits étudiés ;
les Partenaires sélectionnés ;
les critères de comparaison ;
les observations du conseiller ;
les analyses proposées par les Agents IA.

Aucune ressaisie n'est nécessaire.

33. Structure du document

Le document généré comporte une structure homogène.

Exemple :

1. Présentation du Client
identité ;
contexte ;
objectif du Projet.
2. Analyse des besoins
besoins exprimés ;
besoins identifiés ;
contraintes ;
priorités.
3. Étude des solutions
solutions étudiées ;
critères de comparaison ;
solutions écartées.
4. Recommandation
solution retenue ;
justification ;
points de vigilance.
5. Informations réglementaires
documents remis ;
informations communiquées ;
déclarations du Client.
6. Validation
conseiller ;
client ;
dates ;
signatures.

Cette structure peut être adaptée selon le type de Projet.

34. Validation interne

Avant d'être présenté au Client, le document peut suivre un processus de validation interne.

Exemples :

validation par un responsable ;
validation conformité ;
contrôle qualité ;
validation du mandataire.

Les étapes de validation sont paramétrables.

Toutes les validations sont historisées.

35. Présentation au Client

Le document est présenté au Client depuis son espace sécurisé.

Le Client peut :

consulter le document ;
télécharger une copie ;
poser des questions ;
demander des précisions ;
signer électroniquement.

Toutes les consultations sont historisées.

36. Signature

Après validation du Client :

la version est figée ;
un horodatage est enregistré ;
le certificat de signature est associé au document ;
le document est archivé dans le dossier réglementaire.

Toute modification ultérieure entraîne automatiquement la création d'une nouvelle version.

37. Gestion des versions

Chaque version possède :

un numéro ;
une date de création ;
un auteur ;
un statut ;
un motif de création.

Les anciennes versions restent consultables.

La plateforme permet de comparer deux versions successives.

38. Révision

Le devoir de conseil peut être révisé lorsque :

les besoins évoluent ;
le Client modifie son projet ;
un nouveau Produit est étudié ;
une erreur est détectée ;
la réglementation évolue.

Une révision ne modifie jamais une version existante.

Elle crée une nouvelle version liée à la précédente.

39. Archivage

À la clôture du Projet :

la dernière version validée devient la version de référence ;
toutes les versions précédentes sont conservées ;
les justificatifs restent associés ;
les signatures sont archivées ;
les preuves réglementaires sont figées.

L'archivage respecte les durées de conservation applicables.

40. Critères de validation

Le cycle documentaire sera considéré comme conforme lorsque :

le document est généré automatiquement à partir du Projet ;
le conseiller conserve la maîtrise du contenu final ;
les validations internes sont historisées ;
le Client peut consulter et signer le document depuis son espace sécurisé ;
toutes les versions sont conservées ;
les révisions créent systématiquement une nouvelle version ;
le document signé est archivé avec l'ensemble de ses preuves réglementaires.


PARTIE 5 — Le moteur de justification et d'aide à la recommandation
41. Philosophie

Le devoir de conseil ne consiste pas uniquement à produire un document réglementaire.

Il consiste à démontrer le raisonnement ayant conduit à la recommandation.

La plateforme doit donc être capable de reconstruire ce raisonnement à tout moment.

L'objectif est que chaque recommandation puisse être comprise, expliquée et justifiée plusieurs années après sa réalisation.

42. Le moteur de justification

Le moteur de justification constitue un composant central de la plateforme.

Il exploite automatiquement les informations provenant :

du recueil des besoins ;
de la fiche Client ;
des Produits étudiés ;
des Partenaires ;
des analyses du conseiller ;
des comparatifs réalisés ;
des règles métier ;
des Agents IA.

Il transforme ces données en un raisonnement structuré.

43. Les besoins identifiés

Pour chaque recommandation, la plateforme identifie les besoins auxquels elle répond.

Exemple :

Besoin identifié

Réduire le coût de l'assurance emprunteur.

↓

Le Produit retenu permet une économie annuelle estimée de X €.

Besoin identifié

Maintenir une couverture en incapacité.

↓

Le Produit retenu couvre ce risque conformément aux attentes exprimées.

Besoin identifié

Limiter les exclusions médicales.

↓

Le Produit retenu présente des conditions plus favorables que les autres solutions étudiées.

Chaque justification est reliée aux réponses du recueil des besoins.

44. Les garanties associées

La plateforme établit automatiquement un lien entre les besoins du Client et les garanties du Produit.

Exemple :

Besoin du Client	Garantie répondant au besoin
Protection du revenu	Garantie ITT
Protection de la famille	Garantie Décès
Maintien du prêt	Garantie PTIA
Activité professionnelle spécifique	Garantie adaptée à la profession

Le conseiller peut compléter ou corriger ces associations.

45. Les points de vigilance

Le moteur identifie également les éléments nécessitant une explication au Client.

Exemples :

exclusion particulière ;
franchise importante ;
délai de carence ;
limitation de garantie ;
formalités médicales ;
évolution tarifaire connue.

Ces éléments doivent être expliqués avant la souscription.

Ils sont intégrés au devoir de conseil.

46. Les solutions écartées

Le moteur explique pourquoi certaines solutions n'ont pas été retenues.

Exemple :

Produit A

Non retenu car :

garanties insuffisantes ;
absence de couverture recherchée ;
coût supérieur ;
contraintes de souscription.

Produit B

Non retenu car :

exclusions incompatibles avec la situation du Client.

Ces informations restent enregistrées dans le Projet.

47. Les propositions des Agents IA

Les Agents IA peuvent préparer une justification argumentée.

Ils peuvent notamment :

relier les besoins aux garanties ;
détecter les incohérences ;
proposer une rédaction ;
signaler des oublis ;
identifier des produits alternatifs.

Chaque proposition est clairement identifiée comme une suggestion.

Le conseiller conserve la responsabilité de la validation.

48. Le contrôle de cohérence

Avant validation, la plateforme vérifie automatiquement la cohérence de la recommandation.

Exemples :

une garantie recommandée répond-elle réellement à un besoin identifié ?
un besoin important est-il resté sans réponse ?
une exclusion majeure a-t-elle été expliquée ?
une solution pertinente a-t-elle été écartée sans justification ?
les recommandations sont-elles cohérentes avec le profil du Client ?

Les anomalies sont présentées au conseiller avant la validation finale.

49. La justification finale

La plateforme génère automatiquement une synthèse du raisonnement.

Cette synthèse explique :

les besoins exprimés ;
les besoins identifiés ;
les solutions étudiées ;
les critères de comparaison ;
les raisons des choix ;
les raisons des exclusions ;
les points de vigilance.

Le conseiller peut compléter ou modifier cette synthèse avant sa validation.

Elle constitue le cœur du devoir de conseil remis au Client.

50. Critères de validation

Le moteur de justification sera considéré comme conforme lorsque :

chaque recommandation est reliée aux besoins du Client ;
les garanties répondant à chaque besoin sont identifiées ;
les points de vigilance sont explicitement signalés ;
les solutions écartées sont justifiées ;
les Agents IA assistent la préparation sans prendre de décision ;
un contrôle de cohérence est réalisé avant validation ;
le conseiller valide systématiquement la justification finale.
51. Vision à long terme

À terme, ce moteur de justification deviendra l'un des principaux éléments différenciants d'EJ Assurances.

Il permettra :

d'améliorer la qualité des recommandations ;
de renforcer la conformité réglementaire ;
de faciliter les contrôles internes et externes ;
d'accompagner les nouveaux conseillers dans leur montée en compétence ;
d'offrir aux Clients une explication claire et transparente des recommandations formulées.

Le moteur de justification est conçu comme un outil d'aide à la décision, au service du conseiller et du Client.
